## Phase Goal

As a user, I want to purchase seats or cancel reservations without races with the expiration monitor, so that my successful actions are never silently undone.

<objective>
Fix two race conditions in the expiration and cancellation paths:
1. Semaphores released AFTER `table_and_sections` lock drops — seat is AVAILABLE but semaphore unreleased, causing concurrent RESERVE to fail on semaphore acquisition
2. `session.seats` read outside `table_and_sections` lock in `handle_cancel`, `handle_confirm`, and `expire_session` — fragile against concurrent seat additions

Add concurrent stress tests to verify fixes hold under load.

Purpose: Close the timing window between seat-state update and semaphore update. Eliminate the stale `session.seats` snapshot pattern. Verify with stress tests.
</objective>

  One change to `handle_confirm()` in `transactional_thread.py`.

  Move the seat grouping and section ordering INSIDE the `with table_and_sections` block, after the double-check.

  Current code reads `session.seats` outside the lock (line 331: `seats_by_section = self._group_seats_by_section(session.seats)`, line 332: `ordered_sections = self._ordered_sections(seats_by_section.keys())`).

  Move both lines inside the `with` block, after the double-check succeeds:

  ```
  with table_and_sections(ordered_sections):
      current_session = get_by_session_id(session_id)
      if None or not ACTIVE: return failure

      seats_by_section = _group_seats_by_section(session.seats)
      ordered_sections = _ordered_sections(seats_by_section.keys())

      for section in ordered_sections:
          for row, col in seats_by_section[section]:
              verify RESERVED → mark SOLD
      session.state = CONFIRMED
      remove session
  ```

  This ensures the seat list we iterate over matches the state at lock acquisition time, not a snapshot from before acquiring the lock.

  Note: `ordered_sections` is also used as the argument to `table_and_sections()`. Since we now compute it inside the lock, we need to pass `ordered_sections` to `table_and_sections` as before, but we compute it from the seats before entering the lock... 

  Actually wait — we can't move `ordered_sections` inside the lock AND use it as argument to `table_and_sections`. The lock acquisition needs the sections list. However, the sections are determined by the session's existing seats (which don't change section membership — seats are added to the same sections). The actual race is about ADDITIONAL seats being appended. The section SET is correct from the outside read because:
  - Seats can only be added to existing sections (or new sections)
  - If a new section is added, we'd acquire locks for more sections, which is fine (we'd be holding a superset of the needed locks — always safe, never deadlock)

  **Revised approach:** Keep the initial `ordered_sections` computation outside the lock (it's needed for `table_and_sections` argument). But read `session.seats` inside the lock to handle the case where seats were added. Use the same `ordered_sections` from outside (it's a safe superset or exact match) but compute `seats_by_section` from the fresh `session.seats` inside the lock.

  This is simpler and correct: the section lock ordering is determined upfront (safe superset), while the actual seat list is read under lock.
  </action>
