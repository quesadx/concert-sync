Complete the session-TTL migration by making all remaining operations (batch reserve, confirm, cancel, expire) session-aware. CONFIRM and CANCEL now verify session ownership before acting. Expired sessions release all held seats atomically through the MonitorThread.

**Purpose:** After this plan, the entire system operates on session-based TTL. Every operation — RESERVE, RESERVE_BATCH, CONFIRM, CANCEL — goes through SessionManager. TTL expiry happens at the session level. Ownership validation prevents cross-user session interference.

**Output:** Session-aware batch/confirm/cancel handlers, session-level expiry in MonitorThread, TUI TTL tracking fixes.
</objective>

  - Replace the `reservation_table.add_reservation(...)` call with session logic:
    - `session = self.server.session_manager.get_or_create(user_id)`

  - Replace the entire tx_id/reservation table lookup logic:
    - `session_id = request.get("transaction_id")` (the session_id is passed as transaction_id for backward compat)
    - `session = self.server.session_manager.get_by_session_id(session_id)`

    - Set `session.state = ReservationStatus.CONFIRMED`
    - Remove session: `self.server.session_manager.remove(session.user_id)`
    - Return `build_success_response(transaction_id=session_id)`
    - Log: `f"Session:{session_id} User:{session.user_id} confirmed"` in global_log

  **C) `handle_cancel(self, request)` — Session-aware with ownership check:**
  - Identical structure to handle_confirm, but marks seats AVAILABLE and releases semaphores:
    - `session = self.server.session_manager.get_by_session_id(session_id)`
    - Ownership check: `session.user_id == request["user_id"]`
    - Inside section lock: mark ALL seats `SeatState.AVAILABLE`, count released per section
    - Outside section lock: `self.server.semaphore_mgr.release_multiple(section, count)` for each section
    - `session.state = ReservationStatus.CANCELLED`
    - `self.server.session_manager.remove(session.user_id)`
    - Return `build_success_response(transaction_id=session_id)`

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→server | CONFIRM/CANCEL requests carry self-asserted user_id and session_id pairs. Untrusted data crosses here — server MUST verify ownership. |
| MonitorThread→SessionManager | Background thread accessing session state concurrently with TransactionalThreads. Race between confirm and expiry crosses this boundary. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-04 | Tampering | handle_confirm + handle_cancel | mitigate | Ownership check: verify `session.user_id == request["user_id"]`. Return generic `failure_transaction_not_found` on mismatch — do not reveal who owns the session (per RESEARCH.md Pitfall 2 and ASVS V4 Access Control). |
| T-01-05 | Tampering | expire_session vs handle_confirm race | mitigate | `expire_session()` double-checks session state INSIDE the table lock (after acquiring it). If confirm already set state to CONFIRMED, expire_session skips. Confirm handler also acquires table lock — the lock ordering guarantees one wins consistently (per RESEARCH.md Pitfall 3). |
| T-01-06 | Elevation of Privilege | SessionManager.get_by_session_id | mitigate | No privilege system — any user_id can look up any session_id. The ownership check in handle_confirm/handle_cancel provides the actual access control at the operation level. |
| T-01-SC | Tampering | All code in this plan | mitigate | No external packages. Slopcheck to verify no code injection across imports or eval patterns. |
</threat_model>
