<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GUI ↔ ConcertServer (TCP port 9999) | All reservation data crosses via plaintext JSON over TCP sockets — no TLS, no authentication |
| User input ↔ ConcertClient | User-provided host, port, section, row, col, transaction_id flow through ConcertClient to server |
| Worker thread ↔ Main thread | Background threads emit Qt signals carrying server response data to main thread slots |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Spoofing | TCP connection — no TLS | accept | Educational concurrency project; server runs on localhost; TLS would require certificate infrastructure beyond scope per RESEARCH.md §Security Domain |
| T-04-02 | Information Disclosure | Plaintext TCP — seat map data visible to local network | accept | Server binds to localhost only (not 0.0.0.0); traffic never leaves the machine |
| T-04-03 | Tampering | Host/port input fields | mitigate | `_on_connect` validates port as integer; invalid port returns early; ConcertClient handles malformed responses |
| T-04-04 | Spoofing | Session hijacking via user_id | accept | No authentication system exists; educational project per teacher review ("No existe un sistema de login") |
| T-04-05 | Denial of Service | GUI freeze from slow server | mitigate | All network operations run on daemon `threading.Thread` (not main thread); `QTimer` continues polling; `ConcertClient.settimeout` not set but server is local |
| T-04-06 | Elevation of Privilege | User confirms/cancels another user's transaction | mitigate | Server-side ownership check in `handle_confirm`/`handle_cancel` (lines 430-432, 486-488): rejects if `session.user_id != user_id`; client sends own user_id in every request |
</threat_model>
