## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| TCP socket → handle_cancel | Untrusted client sends cancel request with transaction_id — must validate ownership and state |
| TCP socket → handle_confirm | Untrusted client sends confirm request — must verify seats still RESERVED inside lock |
| MonitorThread → SessionManager | Background thread reads session state without full lock — double-check pattern mitigates |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01 | Tampering | handle_cancel double-check | mitigate | Idempotent cancel: if session already CANCELLED inside lock, return SUCCESS instead of raising error; prevents TOCTOU race where two threads pass the pre-lock ACTIVE check |
| T-02-02 | Denial of Service | reclaim_session | mitigate | reclaim_session only remaps user_id under lock; does not create new resources or leak memory; session TTL still applies |
| T-02-03 | Information Disclosure | reclaim_session | accept | session_id is a UUID — guessing is infeasible in practice; no auth system exists in this educational project |
| T-02-04 | Elevation of Privilege | reclaim_session ownership change | mitigate | Only reclaims sessions that are still ACTIVE; does not allow reclaiming CONFIRMED or CANCELLED sessions; original user_id loses access |
</threat_model>
