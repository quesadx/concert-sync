## Design Constraints

1. **UI-03 mandates preservation** — no layout changes, no new panels, no widget replacements. Only seat cell rendering and legend text change.
2. **DataTable cell granularity** — Textual DataTable supports per-cell styling via `update_cell_at(coordinate, value, style=...)`. This is the mechanism for OWN vs OTHER visual differentiation.
3. **No new screens or panels** — all changes are inside `_render_seat_map()` and the legend Static widget.

### Phase 4: Seat State Color Contract (NEW)

The seat map DataTable currently renders all cells in a single color (`#e0f0ff`). Phase 4 introduces **per-state cell styling** via `DataTable.update_cell_at()` with `rich.style.Style` objects. The `_seat_token()` helper is replaced by a method that returns both a token character and a style for each seat.

| State | Token | Text Color | Text Style | Visual Meaning |
|-------|-------|------------|------------|----------------|
| AVAILABLE | `A` | `#e0f0ff` (inherit from table default) | none | Seat is free to reserve |
| RESERVED (by other) | `R` | `#d4a84b` (amber) | **bold** | Seat is temporarily held by another user |
| OWN_RESERVED | `Y` | `#9ad4d6` (teal) | **bold** | Seat is held by YOU in your active session |
| SOLD | `S` | `#556677` (muted) | dim | Seat is permanently purchased |

**Why two letters for reserved?**
- `R` = RESERVED_BY_OTHER (amber bold) — signals "occupied by someone else"
- `Y` = OWN_RESERVED (teal bold) — signals "this is yours" — `Y` mnemonic for "Yours" or "Yes"
- This preserves backward compatibility: existing `R` tokens still mean RESERVED, but own seats now show `Y`
- `A` and `S` tokens unchanged

**Accessibility notes:**
- Teal (`#9ad4d6`) vs amber (`#d4a84b`) provides luminance contrast (~3.5:1 against `#0b1220` background) and hue separation
- Bold text style for both `R` and `Y` adds a shape-independent distinction
- Token characters differ (`R` vs `Y`) — provides text-level redundancy for colorblind users

**Accent (`#9ad4d6`) reserved for:** Panel titles, active borders, OWN_RESERVED seat cells. NOT used for OTHER_RESERVED, SOLD, or AVAILABLE.

## Seat Map Rendering Contract

### Current Implementation (baseline)

```
_seat_token(state) -> str:
  AVAILABLE → "A"
  RESERVED  → "R"
  SOLD      → "S"

_render_seat_map():
  table.add_row(*[_seat_token(s) for s in row], label=f"{row_idx:02d}")
```

### Phase 4 Implementation (target)

Replace `_seat_token()` with a method returning `(token, Style)` pairs:

```
_seat_cell(state: str) -> tuple[str, Optional[Style]]:
  "AVAILABLE"    → ("A", None)                       # inherit table default
  "RESERVED"     → ("R", Style(color="#d4a84b", bold=True))
  "OWN_RESERVED" → ("Y", Style(color="#9ad4d6", bold=True))
  "SOLD"         → ("S", Style(dim=True))
  default        → ("?", Style(color="#ff4444"))

_render_seat_map():
  table.add_row(*[token for token, _ in seat_cells], label=f"{row_idx:02d}")
  for col_idx, (token, style) in enumerate(seat_cells):
    if style is not None:
      table.update_cell_at(Coordinate(row_idx, col_idx), token, style=style)
```

**Key changes:**
1. `_seat_token()` is split into `_seat_cell()` that returns `(token, style_or_None)`
2. Cells are added with their token character, then styled individually via `update_cell_at`
3. `from rich.style import Style` added to imports
4. `from textual.coordinate import Coordinate` added to imports

### Server-Side Prerequisite

`handle_query_seat_map()` in `transactional_thread.py:448` must enrich RESERVED cells with ownership data:

1. Extract `user_id` from the incoming request (already sent by client)
2. Look up the requesting user's active session from `session_manager._sessions`
3. When serializing a seat cell: if state is RESERVED, check if `(section, row, col)` is in the requesting user's session seats. If yes, return `"OWN_RESERVED"` instead of `"RESERVED"`
4. The `SeatState` enum gets a new member `OWN_RESERVED = "OWN_RESERVED"` (or alternatively the server returns the string directly without enum, since the client already maps strings to tokens)

**Return shape change** (backward compatible — existing clients see "RESERVED" for non-owned seats):
```json
{
  "status": "SUCCESS",
  "seat_map": {
    "VIP": [["AVAILABLE", "OWN_RESERVED", "RESERVED", ...], ...],
    ...
  }
}
```

## Flight Path: State & Data Flow

```
User clicks available seat "A"
  → RESERVE request sent with user_id
  → Server marks seat RESERVED, assigns to user's session
  → Next refresh cycle hits QUERY_SEAT_MAP (with user_id in request)
  → Server handle_query_seat_map():
       For each cell:
         if seat == RESERVED AND (section,row,col) in requesting user's session.seats:
           return "OWN_RESERVED"
         else:
           return seat.value (AVAILABLE / RESERVED / SOLD)
  → Client _refresh_query_worker receives seat_map_payload with "OWN_RESERVED"
  → _render_seat_map() maps:
       "OWN_RESERVED" → token "Y", Style(color="#9ad4d6", bold=True)
       "RESERVED"     → token "R", Style(color="#d4a84b", bold=True)
  → Cells rendered with distinct colors
  → Legend shows "Y=YOURS" explanation

When session expires or is confirmed/cancelled:
  → Server removes session (seats revert to AVAILABLE or stay SOLD)
  → No more "OWN_RESERVED" cells for that user
  → Next refresh renders them as "A" (if expired/cancelled) or "S" (if confirmed)
```

## Interaction Contract

| Interaction | Current Behavior | Phase 4 Behavior | Change |
|-------------|-----------------|------------------|--------|
| User reserves a seat | Seat shows "R" in table | Seat shows **"Y" in teal bold** (`#9ad4d6`) | Color + character change |
| User sees others' reservations | Seat shows "R" in table | Seat shows **"R" in amber bold** (`#d4a84b`) | Color change only |
| User cancels/expired | Seat shows "A" in table | Seat shows "A" in table (unchanged) | None |
| User confirms | Seat shows "S" in table | Seat shows "S" dimmed (`#556677`) | Dim style added |
| Legend displayed | `A=AVAILABLE R=RESERVED S=SOLD ...` | `A=AVAILABLE R=RESERVED Y=YOURS S=SOLD ...` | Updated |
| Click on "Y" (own seat) | Shows "State: R" | Shows "State: Y" — explain it's yours | Updated status message |
| Server does not return OWN_RESERVED | Not applicable | Logs warning `[UI] Server did not return ownership data`; falls back to rendering all RESERVED as "R" amber | Graceful degradation |
